# SDD Toolkit Instructions

You are helping a project use Spec-Driven Development (SDD). The core workflow is:

- `/specify` - turn a feature idea into a formal spec
- `/clarify` - refine an existing spec through Q&A
- `/plan` - break a spec into implementation tasks
- `/implement` - generate code from a spec and plan
- `/review` - review code or PR changes against the spec
- `/update` - keep specs and AI memory in sync after code changes

## Core Principles

1. Specs first: requirements go in Markdown before implementation.
2. Traceable: code and tests should map back to requirement IDs.
3. Testable: acceptance criteria should become test cases.
4. Reviewable: PR reviews should compare implementation against the spec.
5. Synced: `/update` keeps specs and project memory aligned with code.
6. Tool-neutral: specs are Markdown and can be used by any AI assistant or developer.

## Workflow

```text
1. /specify "I want a user auth module"
   -> create docs/specs/FEAT-001-auth.md

2. /clarify FEAT-001
   -> refine requirements and acceptance criteria

3. /plan FEAT-001
   -> create docs/specs/FEAT-001-plan.md

4. /implement FEAT-001
   -> generate code and tests matching the spec

5. /review FEAT-001
   -> create reviews/FEAT-001-review.md

6. /update
   -> sync spec status and project memory
```

## Review Guidance

When running `/review`, prioritize:

- Bugs and behavior gaps
- Missing acceptance criteria
- Missing or weak tests
- Security, reliability, and maintainability risks
- Spec drift between requirements and implementation

Findings should be ordered by severity and reference files or requirements when possible.

## Project Context

Use `docs/specs/CLAUDE.md` as the project memory file when present. Treat it as an AI-readable summary of the codebase, current specs, conventions, and decisions.

Use `.github/sdd/` command files as reusable instructions for each SDD command.
